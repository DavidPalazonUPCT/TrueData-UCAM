# ADR-003: Pipeline v2 — Node-RED como pre-procesador con Gateway MQTT API

- **Status:** Proposed (pendiente de PoC y validación con OPC client vendor/AI team)
- **Date:** 2026-04-15
- **Autor:** David Palazon (Base module team)
- **Contexto previo:**
  - [ADR-001](ADR-001-current-pipeline.md) — arquitectura actual (5 hops HTTP, fan-out en NR)
  - [ADR-002](ADR-002-aggregation-store-forward.md) — preguntas abiertas sobre ventanas y store-and-forward

## 1. Contexto y motivación

El análisis del pipeline actual (ADR-001) y del dataset FR_ARAGON (ADR-002)
identificaron dos problemas fundamentales en la arquitectura de ingestión:

### 1.1 El problema de la alineación temporal y la agregación inútil

> **Nota (2026-04-16):** esta sección estaba originalmente titulada
> *"El problema del jitter intra-scan"* con una atribución incorrecta
> del origen del jitter al OPC Client. La evidencia empírica del dump
> source (ADR-002 §3.5) obligó a reescribirla. La decisión del refactor
> v2 no cambia; lo que cambia es el diagnóstico.

El OPC Client implementa subscriptions OPC-UA estándar (Part 4 §5.12,
§5.13): cada `DataChangeNotification` es un bundle de N valores
(1 ≤ N ≤ 27) con un único `sourceTimestamp` del PLC compartido por
todos los valores del bundle. Esto está **confirmado empíricamente**
en el dump `connect_opcua_itemvalue` (ADR-002 §3.5.1): 10 840 bundles
con 27 tags al microsegundo idéntico, sin jitter intra-bundle.

El pipeline v1 rompía esa alineación aguas abajo:

- NR recibía el bulk del OPC Client y hacía **fan-out secuencial**
  con 27 peticiones HTTP separadas a TB (una por tag, con token
  individual). Cada petición se procesaba con wall-clock distinto →
  TB almacenaba los 27 valores con `ts` levemente desalineados
  (~1-3 ms de dispersión). Este **era el origen real del jitter
  observado en TB**, no el OPC Client.
- Ante ese "jitter aparente", el pipeline v1 intentaba re-alinear
  con ETLflows (ventanas 1s/5s/10s) para reconstruir scans. Pero
  con cadencia 34 s, cada ventana contenía 0 o 1 sample → la
  agregación degeneraba en identidad. Además, los nulos espurios
  aparecían cuando tags desalineados caían en ventanas distintas,
  reintroduciendo el problema que la ventana pretendía resolver.
- Truncar al segundo tampoco resolvía el edge case de cruce de
  frontera (~0,3 % por scan, ~52 scans afectados en 30 días de
  FR_ARAGON).

La causa raíz era el fan-out. V2 lo elimina: NR deja de fan-outear
y publica 1 mensaje MQTT Gateway con `ts` único del PLC replicado
a todos los tags del bundle. La alineación se preserva por
construcción, sin ventanas, sin re-alinear nada.

### 1.2 El problema del fan-out con tokens

El patrón Raw Device + fan-out en Node-RED requiere:

- Un diccionario `global.diccionarioAccess` con tokens de todos los
  devices, sin invalidación
- N peticiones HTTP (una por sensor) con token en URL path (limitación de
  la API device HTTP de TB CE)
- 5 hops HTTP total, con Node-RED como single point of failure en el path
  crítico

### 1.3 La oportunidad: Gateway MQTT API

ThingsBoard CE 4.x ofrece la [Gateway MQTT API](https://thingsboard.io/docs/reference/gateway-mqtt-api/)
que permite:

- Un único device tipo "Gateway" con **un solo token**
- Publicar telemetría de **N devices en un solo mensaje MQTT**
- **Auto-provisioning** de sub-devices (si no existen, TB los crea)
- Auth via MQTT username (token no viaja en URL)
- Fan-out nativo dentro de TB (sin hops externos)

## 2. Decisión

Reposicionar Node-RED **delante** de ThingsBoard (entre el OPC Client y TB),
en lugar de detrás, y usar la Gateway MQTT API como transporte de NR a TB.

### 2.1 Arquitectura v2

       │
       │ OPC-UA
       ▼
┌──────────────────────────────────┐

### 2.2 Comparativa directa con v1

| Aspecto | v1 (ADR-001) | v2 (esta propuesta) |
|---|---|---|
| Posición de NR | Detrás de TB (post-procesador) | Delante de TB (pre-procesador + router) |
| Fan-out a TB | NR: 27 HTTP POSTs con tokens individuales | TB nativo via Gateway MQTT API |
| Fan-in para ML | ETLflows: ventanas → bucket devices → modelo lee de TB | **NR directo:** el dato llega junto, se envía junto |
| Tokens necesarios | 1 (raw) + N (per-sensor) + 6 (buckets) | **1** (gateway device) |
| Diccionario de tokens en NR | Sí (`global.diccionarioAccess`) | **No** |
| Hops OPC→telemetría almacenada | 5 HTTP | **2** (HTTP OPC→NR + MQTT NR→TB) |
| Hops OPC→ML inference | 5+ (OPC→TB→NR→TB→NR→TB→ML) | **2** (OPC→NR→ML) |
| Resolución de jitter | Ventanas wall-clock en ETL (buggy) | **No aplica** — 1 POST = 1 ts |
| Capa de agregación | ETLflows + 6 buckets + profile + rule chain | **Eliminada** |
| Timestamp | Ambiguo (mezcla server/client side) | **Client-side del PLC** (siempre) |
| Cambios en el OPC client | — | **Cero** |
| NR como SPOF en ingestión | Sí (fan-out pasa por NR) | Sí (sigue en el path) |
| Store-and-forward | No hay | **Pendiente** (ver ADR-002 §2.2) |

### 2.3 Decisiones de diseño clave

#### 2.3.1 Client-side timestamp (del PLC), siempre

El timestamp del PLC responde a "¿cuándo ocurrió la medición en la planta?".
El timestamp de TB responde a "¿cuándo lo procesó el sistema?". Para anomaly
detection, la primera pregunta es la relevante.

Además, client-side ts es requisito para que store-and-forward funcione:
si datos retrasados llegan a TB con server-side ts, toda la historia
temporal se colapsa en "ahora". Con client-side ts, los datos retrasados
conservan su posición temporal real.

TB almacena el timestamp de ingestión como metadata interna, así que la
segunda pregunta sigue siendo respondible sin sacrificar la primera.

#### 2.3.2 Alineación temporal preservada por diseño (no requiere debounce)

> **Nota (2026-04-16):** redactado originalmente como *"jitter intra-scan
> resuelto por diseño"*. El dump source demostró que el jitter intra-bundle
> en origen es cero (ADR-002 §3.5), así que lo que v2 "resuelve por
> diseño" no es un jitter del OPC Client sino la rotura de alineación
> que v1 introducía en el fan-out. Reescrito en consecuencia.

El OPC Client emite `DataChangeNotification`s OPC-UA con un único
`sourceTimestamp` compartido por todos los valores del bundle (ADR-002
§3.5.1, Part 4 §7.7.3). El problema operativo real era que **v1 rompía
esa alineación en el fan-out** (27 POSTs HTTP secuenciales → 27 `ts`
levemente distintos en TB). Se evaluaron tres soluciones para el
pipeline v2:

- **Truncar ts al segundo:** mitigación cosmética que no aborda la
  causa. Además, reintroduciría el edge case de cruce de frontera
  (~0,3 %/scan). Descartado.

- **Debounce 100 ms (join con timeout):** agrupar tags por proximidad
  temporal. Tendría sentido si los tags llegaran como mensajes
  separados a NR, pero ese no es el caso en v2. Fue el plan inicial
  antes de confirmar que el OPC Client envía un bulk.

- **Preservar el `sourceTimestamp` de origen (solución adoptada):** el
  OPC Client postea un único HTTP POST por `DataChangeNotification`
  (1 bundle = 1 ts = 1..N tags). NR extrae el `ts` del payload y lo
  replica idéntico a todos los tags al construir el mensaje Gateway
  MQTT. Al eliminar el fan-out secuencial, la alineación que el PLC
  asignó llega intacta a TB. Cero jitter por construcción porque NR
  nunca ve los tags por separado.

La tercera solución es la más simple y robusta: cero nodos join, cero
timers, cero state. El function node de NR tiene ~30 líneas. Si un
futuro cliente enviase tags en POSTs separados, el debounce join se
puede reintroducir sin cambiar la arquitectura.

**Nota sobre cardinalidad variable:** el OPC Client también emite
notificaciones aisladas entre publishing cycles completos — por
ejemplo, `Heart_Bit` (watchdog) o analógicos sin deadband que cambian
más a menudo que el ciclo nominal (ADR-002 §3.5.1). En v2 cada una de
esas notificaciones es un POST válido con 1 tag y `ts` propio. El
contrato `/api/opc-ingest` acepta cardinalidad 1..N sin validación
(PLAN-001 §D.2.2), por lo que el patrón bimodal del OPC Client llega
a TB sin transformaciones.

#### 2.3.3 NR sigue en el path crítico (aceptado)

NR pasa de post-procesador a pre-procesador, pero sigue siendo SPOF. Esto
es aceptable porque:

- NR ya estaba en el path crítico en v1 (fan-out dependía de NR)
- La diferencia es que ahora NR hace **menos** (parse + build payload
  vs split + token lookup + 27 HTTP POSTs)
- Menos código = menos superficie de fallo
- El store-and-forward (ADR-002 §2.2) debería estar en el OPC Client
  (antes de NR), no en NR. Si NR cae, el OPC Client encola. Cuando NR
  vuelve, los datos fluyen con sus ts originales del PLC.

## 3. Qué se elimina del pipeline v1

| Componente eliminado | Por qué |
|---|---|
| Raw Device + `raw_root_rule_chain` | Ya no hay device receptor bulk; TB recibe per-device via Gateway |
| `PrevioRawFlow.json` (NR fan-out) | El fan-out es nativo en TB via Gateway MQTT |
| `global.diccionarioAccess` en NR | Un solo token (gateway), no N tokens |
| `profile_raw` (device profile) | Sin Raw Device, el profile sobra |
| `ETLflows.json` (ventanas 1s/5s/10s) | Jitter resuelto por construcción; agregación inútil a 34s; fan-in para ML resuelto en NR |
| `profile_buckets` + 6 bucket devices | Sin ETLflows, los buckets no tienen fuente de datos |
| `buckets_root_rule_chain` | Sin buckets, sin rule chain |
| `TbRestApiCallNode` en `devices_root_rule_chain` (call a NR ETL) | Sin ETLflows, el call sobra |

### 3.1 Qué se mantiene sin cambios

| Componente | Por qué se mantiene |
|---|---|
| Critical Levels flows (NR) | Lógica de alarmas separada del path de ingestión |
| devices_root_rule_chain | Rule chain per-device sigue necesaria (sin el TbRestApiCallNode de ETL) |
| Deploy pipeline (`env_client.py`) | Se simplifica (menos entidades) pero no desaparece |
| OPC Client HTTP (vendor) | **Cero cambios** — solo cambia quién escucha |
| ML inference API | **Sin cambios** — recibe el mismo body, ahora directo de NR en vez de leer de TB |
| Blockchain service | **Sin cambios** — sigue recibiendo de ML inference |

### 3.2 Qué cambia

| Componente | Cambio |
|---|---|
| Docker Compose | NR expone endpoint HTTP para OPC Client; NR conecta MQTT a TB broker |
| NR flows | Nuevo flow: HTTP-in → parse + build → MQTT-out + HTTP-request ML (5 nodos, 2 salidas paralelas) |
| TB | Crear 1 device tipo Gateway; configurar auto-provisioning |
| devices_root_rule_chain | Quitar TbRestApiCallNode que llamaba a NR para ETL |
| Deploy pipeline | Quitar provisioning de: Raw device, profile_raw, profile_buckets, 6 bucket devices, buckets_root_rule_chain. Añadir: Gateway device + device profile `sensor_planta`. **No se provisionan los 27 devices per-sensor** (se auto-crean via Gateway `connect` con `type`, ver §4.2) |

## 4. Preguntas abiertas que bloquean la implementación

### 4.1 Contrato de la API de ML inference

NR va a POSTear directamente al servicio ML con los 27 sensores. Para
implementar el function node, necesitamos saber:

- **URL del servicio** dentro de la red Docker (hostname:puerto)
- **Formato del body** que espera `/api/inference` (¿`{ts, sensors: {...}}`?
  ¿array? ¿otro?)
- **¿Espera todos los tags o un subconjunto?** Si el modelo solo usa 18 de
  27 tags (excluye las 13 digitales constantes + heartbeat), NR puede filtrar
- **¿Qué devuelve?** No afecta al flow de NR (la respuesta la gestiona
  el propio servicio ML → TB), pero conviene documentar

### 4.2 Auto-provisioning de devices — RESUELTO (2026-04-16)

**Hallazgo (análisis doc TB-CE 4.1.0):** el topic `v1/gateway/connect`
acepta un campo opcional `type` (nombre del device profile). Si el
profile no existe, TB lo crea; si el device no existe, TB lo crea **con
ese profile** (no con el default del tenant). Ver
[Gateway MQTT API — Device connect](https://thingsboard.io/docs/reference/gateway-mqtt-api/#device-connect-api).

El topic `v1/gateway/telemetry`, en cambio, **no admite** `type`: si el
device no existe, se crea con el profile `default`. Esto significa que
el control de profile solo es posible via `connect`.

**Opciones reales reevaluadas:**

| Opción | Pre-provisioning en deploy | Mantenimiento | Veredicto |
|---|---|---|---|
| **A — NR envía `connect` con `type` on-first-seen, lazy** | Solo el device profile | Automático; auto-heal tras restart de NR | **Recomendada** |
| B — NR envía `connect` para los 27 devices al boot | Solo el device profile | Estático; si el PLC expone un tag nuevo, redeploy NR | Alternativa aceptable si se prefiere determinismo |
| C — Pre-crear los 27 devices via REST API en deploy pipeline | Profile + 27 devices por cliente | Deploy pipeline toca 1+27 entidades | **Descartada** — 27 entidades redundantes y escalado lineal con el número de clientes |
| D — Cambiar el default device profile del tenant a `sensor_planta` | Solo el profile, pero afecta a **todo** el tenant | Simple | Descartada — blast radius sobre cualquier device futuro del tenant |

**Recomendación:** opción A.

- `connect` es idempotente por diseño. NR mantiene un `Set<string>` de
  devices ya conectados en memoria del flow; al arrancar se vacía y se
  repuebla bajo demanda cuando llega el primer tag de cada sensor
- Deploy pipeline no provisiona los 27 devices: solo el Gateway device
  y el device profile `sensor_planta`
- Si el PLC expone un tag nuevo en el futuro, el pipeline v2 lo absorbe
  sin cambios de deploy: OPC Client lo envía → NR lo connecta → TB lo
  crea con el profile correcto
- Consecuencia indirecta: las rule chains del device profile y los
  Calculated Fields del profile aplican automáticamente al nuevo device

**Riesgo residual:** si NR muere justo entre `connect` y `telemetry` del
mismo scan, el device queda creado pero sin telemetría en ese scan. No
es un error acumulable — el siguiente scan lo resuelve. Aceptable.

**Impacto en PLAN-001:**
- Fase 2 añade un paso de `connect` idempotente antes del `telemetry`
- Fase 3 pierde el trabajo de provisionar los 27 devices: solo crea
  Gateway device + device profile `sensor_planta`

## 5. Riesgos

| # | Riesgo | Mitigación |
|---|---|---|
| R1 | NR como pre-procesador es SPOF para toda la ingestión | Store-and-forward en OPC Client (ADR-002 R2). Si NR cae, OPC encola |
| R2 | Futuro cliente envía tags en POSTs separados (no bulk) | Reintroducir join debounce 100ms en NR. Arquitectura no cambia |
| R3 | Gateway MQTT API tiene quirks no documentados en CE 4.1 | **MITIGADO (2026-04-16):** PoC Fase 0 ejecutada con éxito, 4/4 criterios validados: auto-provisioning de devices, timestamps client-side preservados, QoS 1 con PUBACK, rule chain del device profile ejecutándose. Ver [PLAN-001 §Resultado Fase 0](PLAN-001). El comportamiento de la Gateway MQTT API coincide con la documentación oficial |
| R4 | ETLflows deja de recibir datos al cambiar el path de ingestión | Verificar cómo ETLflows se activa (§4.1) antes de eliminar el path v1 |
| R5 | Deploy pipeline necesita refactor para provisionar Gateway device | Cambio incremental: añadir Gateway, no romper lo existente |

## 6. Alternativas consideradas

| Alternativa | Descartada por |
|---|---|
| **Truncar ts al segundo** | Edge case de cruce de frontera (~52 scans/mes partidos). Innecesario: con 1 POST bulk, NR asigna un ts único a todos los tags sin manipulación temporal |
| **Debounce join 100ms** | Innecesario: el OPC Client envía 1 POST = 1 scan. El debounce solo sería necesario si los tags llegaran en POSTs separados. Se reserva como plan B para futuros clientes |
| **TB asigna server-side timestamp** | Rompe store-and-forward: datos retrasados pierden temporalidad real |
| **OPC Client publica MQTT directo** (sin NR) | El OPC client vendor tiene HTTP implementado, cambio de transporte sin justificación |
| **TBEL script en raw_root_rule_chain** (fan-out sin NR) | Mantiene el patrón Raw device; TBEL limitado para split dinámico |
| **tb-gateway container** (IoT Gateway oficial) | Scope change para el OPC client vendor; rechazado por el equipo Base |
| **Scheduler periódico** (enviar cada N s fijo) | Introduce latencia artificial; el PLC es la fuente de verdad temporal |
